## Observations

- **SO = 0 across the board** is expected: all four trays sit in the same
  NVL72 NVLink fabric domain, so DeepEP V2 routes the entire dispatch and
  combine traffic over NVLink (SU). To get non-zero SO numbers we need a
  pool that spans NVL72 boundaries (multi-rack RDMA hops).

- **NVLink scaling, dispatch SU**:

  | tokens | EP=4 max  | EP=8 max  | EP=16 max | drop 4→8 | drop 8→16 |
  | -----: | --------: | --------: | --------: | -------: | --------: |
  |   1024 |  632 GB/s |  598 GB/s |  564 GB/s |    −5.4% |     −5.7% |
  |   2048 |  745 GB/s |  685 GB/s |  648 GB/s |    −8.1% |     −5.4% |
  |   4096 |  811 GB/s |  726 GB/s |  679 GB/s |    −10.5%|     −6.5% |
  |   8192 |  849 GB/s |  745 GB/s |  699 GB/s |    −12.2%|     −6.2% |

  Larger tokens push closer to GB300 NVLink line rate (~900 GB/s);
  per-doubling of EP costs roughly 6-12 % SU due to additional fabric
  hops and switch contention. The combine path tracks dispatch within
  ~5 %.

- **Local copy bandwidth** stays in the 5.5-7.6 TB/s range across all
  shapes, confirming HBM staging is not the bottleneck.

## Reproduce

Same head_tray + tray pool, all 6 stages can be skipped except the
sweep itself:

```bash
SKIP_SSH_SETUP=1 SKIP_PIP=1 SKIP_BUILD=1 \
  HEAD_TRAY=pod4-gb300-2-tray06-f3 \
  TRAYS="pod4-gb300-2-tray06-f3 pod4-gb300-2-tray07-f3 pod4-gb300-2-tray08-f3 pod4-gb300-2-tray09-f3" \
  bash scripts/jumphost_deepep_sweep.sh
```

For a brand new tray subset (any 4-of-18 NVL72), drop the three `SKIP_*`
flags. See `docs/gb300_setup.md` for full provisioning details.
