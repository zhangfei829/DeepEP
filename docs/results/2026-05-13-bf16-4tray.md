## Observations

- **NVLink scaling (dispatch, topk=8, BF16)** — kernel max GB/s:

  | tokens | EP=4 | EP=8 | EP=16 |
  | -----: | ---: | ---: | ----: |
  |      1 |    2 |    3 |     4 |  ← latency-bound, payload too small
  |     16 |   27 |   34 |    31 |
  |     64 |  107 |  141 |   147 |
  |    128 |  191 |  242 |   249 |
  |    512 |  491 |  479 |   457 |
  |   4096 |  810 |  724 |   679 |

  Saturates around 800 GB/s for EP=4 / tokens=4096, drops ~12% per EP doubling
  due to additional fabric hops + switch contention.

- **kernel vs api gap (dispatch, tokens=4096, topk=8)**:

  | EP | kernel us avg | api us avg | host overhead |
  | -- | ------------: | ---------: | ------------: |
  |  4 |       137.2   |      236.9 |       **+100 us / +73%** |
  |  8 |       225.6   |      349.7 |       +124 us / +55% |
  | 16 |       296.5   |      432.0 |       +135 us / +46% |

  The host wall-clock adds ~100-135 us of fixed overhead per dispatch call.
  Larger payloads amortize this, smaller payloads pay it as pure cost: at
  tokens=1, kernel us ≈ 15 us but api us ≈ 150 us — **10× host overhead**.
  This is the column that matches the "API GB/s" used by Hybrid_ep / NCCL EP
  reports.

- **warps/block** is constant for this matrix:
  - `dispatch = 12 = 4 notify + 8 dispatch` (64 SMs × 12 warps × 32 = 24,576 threads)
  - `combine  = 16 = 16 warps`              (64 SMs × 16 warps × 32 = 32,768 threads)

- `SO/RDMA` traffic is zero across the entire matrix because all 16 ranks
  share one NVL72 NVLink domain. The Scale-Out path is exercised only when
  the EP world straddles ≥2 NVL72 fabrics.
